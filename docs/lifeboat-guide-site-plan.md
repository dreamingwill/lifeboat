# 计划：骇浪求生攻略网站（拆文件版，分模块执行）

## 目标
将 `docs/game_guide_data.md` 的完整攻略内容整理为一个可交互的静态网站，支持直接打开 `index.html` 使用，并可部署到 Cloudflare Pages。

## 定位说明
- 这是一个面向攻略站的新版本方案，不再强制维持“所有逻辑都在单个 `index.html` 中”的旧结构。
- 最终产物仍然不依赖构建工具、包管理器、外部 CDN 或前端框架。
- 页面入口仍为 `index.html`，但允许拆分为本地 CSS / JS 文件，提升可维护性。

## 交付文件
- `index.html`：页面结构、锚点、挂载点
- `styles.css`：全局样式、布局、Hero、响应式、组件样式
- `data.js`：结构化内容数据与角色数据
- `app.js`：页面渲染逻辑与轻交互逻辑

可选后续拆分：
- `score-simulator.js`：计分模拟器逻辑
- `widget.js`：角色 widget 单独封装

## 保留与约束
- 保留 `lifeboat_game_guide.svg`，但默认不引用它作为页面成立的前提
- 不使用外部 CDN、外部字体、外部框架
- 攻略正文内容来源于 `docs/game_guide_data.md`
- 角色数据来源于现有 `index.html` 的 `chars[]`
- 页面可直接通过浏览器打开，不依赖构建工具或本地服务器
- 如需兼顾未来嵌入场景，样式层优先保留 CSS 变量入口，避免把主题写死

## 数据分层建议

### `data.js`
负责：
- `chars`
- 静态 section 内容数据
- FAQ 数据
- 变体 / 扩展数据
- 策略标签页数据
- 计分规则与角色加成映射

建议结构：
```js
const siteMeta = {
  title: '骇浪求生 Lifeboat 攻略站',
  sections: []
};

const overviewData = {};
const setupData = {};
const flowData = {};
const mechanicData = {};
const scoringRules = {};
const chars = [];
const strategyTabs = [];
const variantCards = [];
const faqItems = [];
```

建议原则：
- `data.js` 只放数据，不放 DOM 操作
- `app.js` 只负责渲染与交互，不手写大段静态文案
- 复杂逻辑优先单独封装函数，避免全部堆进一个初始化函数中

## 推荐执行路线
如果按最省返工的方式推进，建议顺序为：

1. 先完成模块 1，只搭建站点骨架与 section 锚点
2. 立刻完成模块 4，优先迁移现有角色 widget，尽快保住已有成果
3. 再做模块 3，把 `docs/game_guide_data.md` 中的主体内容映射进页面
4. 然后完成模块 2，统一视觉与响应式
5. 最后再做模块 5 和模块 6，处理计分器与轻交互
6. 用模块 7 做最终验收和收口

这样做的原因：
- 角色系统是当前仓库里最确定、最容易复用的部分
- 计分模拟器是逻辑最复杂的部分，应放到页面结构稳定后实现
- 先看到“可打开、可浏览、可用”的版本，比先做复杂逻辑更稳

## 实际使用方式
这份计划建议按“任务清单”而不是“说明文档”来使用：

1. 开工前，先把当前要做的模块标记为唯一目标，避免多模块并行导致返工。
2. 每完成一个模块，就直接在浏览器打开 `index.html` 做一次人工检查。
3. 新增内容时优先改 `data.js`，样式问题优先改 `styles.css`，交互问题优先改 `app.js`。
4. 只有当页面结构稳定后，再引入计分模拟器和更细的交互。
5. 如果某个模块做完后需要大改前一模块，说明边界定义不够清晰，应先回到计划修正模块范围。

## 风险提醒
- `docs/game_guide_data.md` 内容很多，如果不先做结构化整理，`app.js` 会很快失控。
- 如果一开始就做计分模拟器，容易在规则细节上反复返工。
- 如果角色 widget 不先迁移，后续很容易出现两套角色数据并存的问题。
- 如果样式完全写死，未来嵌入到其他页面时会失去当前项目“可继承主题”的优势。
